Microsoft Visual Studio .NET

Setup Administrator Mode Readme

1. Introduction

Microsoft Visual Studio .NET offers an administrator mode setup, which facilitates deployment to field clients. The administrator initially runs setup to select which features will be deployed. An .INI file is created based on this selection, and is then deployed by running setup in unattend mode, pointing to the generated .INI file. When run in unattend mode, setup runs silently installing the designated options. If there are any errors, setup exits silently with failure. This allows deployment through Microsoft Systems Management Server.

For information on deploying Visual Studio .NET using SMS, see http://support.microsoft.com/support/kb/articles/Q309/6/57.asp.

2. Running Administrator Mode Setup

Before running this phase of administrator setup, you must first run Windows Component Update setup on your machine.  Some of the components installed by the update are required for Visual Studio .NET setup to function.

Note: You must run the setup.exe found in the \setup subdirectory in order to create an .ini file. The setup.exe file found at the root of the CD or DVD does not support administrator setup.

To create the deployment .INI file
  1. Insert the Visual Studio .NET CD 1 or DVD and ignore the autorun message.
  2. Choose the Start menu and then choose Run.
  3. Type the following:

    n:\setup\Setup.exe /createunattend <path to .INI file>

    For example:

    D:\setup\setup.exe /createunattend c:\vs7_deployment.ini

To run setup with the generated file on a client
  1. Verify that Visual Studio .NET is available on the network (see Creating a Network Image).
  2. Choose the Start menu and then choose Run.
  3. Type the following:

    \\<computer>\<share>\...\setup\Setup.exe /unattendfile <path to .INI file>

    For example:

    \\products\VisualStudio\setup\setup.exe /unattendfile \\computer\share\vs7_deployment.ini

3. Creating a Network Image

Administrator setup is designed to run from a network share, rather than from CDs. The following procedures illustrate how to prepare Visual Studio .NET for installation from a network share.

To share the installation files from the CDs
  1. Create a top-level folder (for example, VS7) on the server.
  2. Create a folder named WCU within the top-level folder (for example, VS7\WCU).
  3. Copy the contents of the CD labeled Visual Studio .NET Windows Component Update to the folder named WCU.
  4. Create a folder named VS within the top-level folder.
  5. Copy the contents of the remaining Visual Studio .NET CDs to the VS7\VS folder (choose to overwrite existing files).
    Note   Optionally, you may also copy the contents of other CDs, such as SourceSafe, Visio, and so on, to separate folders for setup over a network.
  6. Open setup.ini in the VS folder using a text editor, such as Notepad.
  7. In the [Baseline] section, change the line that starts with "DIR=" to "DIR=..\WCU". For example:
    [Baseline]
    DIR=..\WCU
    FileName=setup.exe
  8. Save your changes and close the file.

    Completing this step prevents disk-swapping requests during setup and informs setup of the correct path to the Windows Component Update disk.

  9. Open the folder containing the copied files, and then open the Setup folder.
  10. Open the file "setup.sdb" in a text editor, such as Notepad.
  11. At the end of the file, add the following:

    [Product Key]
    <25 digit Product Key>

    Note: The Product Key is found on the CD packaging.
  12. Save the changes and close the file.
  13. Create a network share to the VS7 folder and establish the appropriate security settings. The path to Visual Studio .NET setup from the network is now:\\[servername]\VS7\VS\setup.exe
    Note   When installing, setup fails if any path and file name combination exceeds 260 characters. The maximum length of a path in Visual Studio is 221 characters; accordingly, you should copy files to a path with less than 70 characters. If you create a network share for a network image, the UNC path to the root install location should contain fewer than 39 characters. For example:

    Not valid: \\products\corporate\licenses\DeveloperTools\Visual Studio\Visual Studio .NET\NetSetup
    Valid: \\products\DeveloperTools\Visual Studio .NET\NetSetup\
To share the installation files from the DVD
  1. Copy the contents of the DVD into a top-level folder named VS7.
    Note   Optionally, you may also copy the contents of other CDs, such as SourceSafe, Visio, and so on, for setup over a network.
  2. Open the folder containing the copied files, and then open the Setup folder.
  3. Open the file "setup.sdb" in a text editor, such as Notepad.
  4. At the end of the file, add the following:

    [Product Key]
    <25 digit Product Key>

    Note: The Product Key is found on the CD packaging.
  5. Save the changes and close the file.
  6. Create a network share to the VS7 folder and establish appropriate security settings. The path to Visual Studio .NET setup from the network is now: \\[servername]\VS7\setup.exe
    Note   When installing, setup fails if any path and file name combination exceeds 260 characters. The maximum length of a path in Visual Studio is 221 characters; accordingly, you should copy files to a path with less than 70 characters. If you create a network share for a network image, the UNC path to the root install location should contain fewer than 39 characters. For example:

    Not valid: \\products\corporate\licenses\DeveloperTools\Visual Studio\Visual Studio .NET\NetSetup
    Valid: \\products\DeveloperTools\Visual Studio .NET\NetSetup\

4. Installation Maintenance

Setup can be run again on a machine with an existing installation. This should be done by the administrator running in createunattend mode on a master machine that had the same install as the client machines. Changes to the feature state can then be deployed to clients.

Clients should not run setup from Add/Remove Programs in Control Panel (or any other means) on their own. This will cause a discontinuity in the feature state. All changes should come from setup being run with the unattendfile switch.

6. Known Installation Issues

6.1. Data file cannot be created

When running setup in createunattend mode with a file that already exists, you will see an error "The data file <filename> cannot be created. Setup cannot continue." Either delete the existing file, or provide a unique file name.

7. Deployment of Visual Studio .Net Service Releases

The deployment of a Visual Studio Update is a two phase process. In the first phase, you download and save the service releases from the Internet or from disks to a local drive. In the second phase, you deploy the service releases to target machines in one of three ways: by disks, a network share, or forced deployment using SMS. Deploying via disks or network shares requires that users initiate the update manually. Deploying via a network share also requires registry changes on the target machines.

Some additional items:

  • For any of these scenarios, you can use UNC paths, absolute paths, or http paths.
  • From a given list of Service Releases, you can freely choose to download any combination of a single available Service Pack and/or a single available Windows Component Update per session.
  • Windows Component Updates are locale specific and restricted to the locale of the operating system of the machine on which the download is performed. For example, if you want download an English Windows Component Update, use a machine with an operating system whose Locale is English.
  • For successful deployment, make sure that all the internal and external dependencies are correctly enforced on the target machines. For example, a particular Service Pack is dependent on a particular Windows Component Update which in turn is dependent on a newer version of Internet Explorer. In order to deploy the Service Pack successfully, you must first install the correct version of Internet Explorer on the target machines, followed by Windows Component Update, before deploying the Service Pack.

Downloading Service Releases

You can copy service releases from two different locations: the Internet or from disk.

To download and copy Service Releases from the Internet
  1. From the Start menu, choose Run.
  2. In the Open box, enter the following to launch Administrator Mode of Visual Studio Service Release:

    "n:\...\vsupdate.exe" /mode=admin /locale=localecode

    For example:

    c:\vsupdate.exe /mode=admin /locale=1033

    Note: If the localecode is omitted, Visual Studio Update defaults to the locale used by the version of Visual Studio installed. If Visual Studio Update cannot find the Visual Studio locale, the locale of the operating system is used. If the operating system locale cannot be determined, English is used by default. See section 9 for a list of locale codes.
  3. Choose Check for Service Releases on the Internet.
  4. Select a Service Release from the list, and then select Web from the Download from drop down menu.
  5. Choose Download Now!.
  6. When prompted, specify a location to save the files.

    Note: Some components may have dependencies. For example, a Service Pack may require Windows Component Update or a newer version of Internet Explorer. If such a dependency exists, a message will appear, giving you the option to download the dependency from the Internet or copy it from disk. If you do not wish to download or copy the dependencies now, you can ignore the warnings and proceed with downloading the Service Pack by choosing continue.

You can also copy Service Releases that you have received on CD.

To copy Service Releases from disk
  1. From the Start menu, choose Run.
  2. In the Open box, enter the following to launch Administrator Mode of Visual Studio Service Release:

    "n:\...\vsupdate.exe" /mode=admin /locale=localecode

    For example:

    c:\vsupdate.exe /mode=admin /locale=1033

    Note: If the localecode is omitted, Visual Studio Update defaults to the locale used by the version of Visual Studio installed. If Visual Studio Update cannot find the Visual Studio locale, the locale of the operating system is used. If the operating system locale cannot be determined, English is used by default. See section 9 for a list of locale codes.
  3. Insert the media containing the Service Release into your CDROM or DVD drive.
  4. Choose Copy a Visual Studio .NET Service Release from disk.
  5. Browse to the drive containing the Service Release media and choose Open.
  6. Select the Service Release from the list and choose Download Now!.
  7. When prompted, specify a location to save the files.

    Note: Some components may have dependencies. For example, a Service Pack may require Windows Component Update or a newer version of Internet Explorer. If such a dependency exists, a message will appear, giving you the option to download the dependency from the Internet or copy it from disk. If you do not wish to download or copy the dependencies now, you can ignore the warnings and proceed with downloading the Service Pack by choosing continue.

Deploying Service Releases

You can deploy Service Release using three different methods: forced deployment with SMS, disks, or network shares.

Forced deployment via SMS

  • Dependencies for the Service Release must be installed on the target machines.
  • All hot fixes, or QFEs, must be removed from the target machines. To remove a hot fix, use Add/Remove Programs in Control Panel.

For more information, see the SMS documentation.

To force updates on target machines using SMS
  1. From the Start menu, choose Run.
  2. In the Open box, enter the following:

    msiexec /p "n:\...\myfile.msp" novsui=1

    For example:

    msiexec /p "\\vsupdate\sr1.msp" novsui=1

Deploying via disks

  • Target users must have administrator privileges on their respective machines.
  • Dependencies for the Service Release must be installed on the target machines.
To force updates on target machines using SMS
  1. Browse to the location where the Service Release files were downloaded or copied.
  2. Copy the files to disk or burn the files to CD.

    Tip: To conserve space, you can delete all of the cabs directories and their contents. Cabs are required for Web deployment only.

Deploying from a shared location

  • Target users must have read permissions to the shared location.
  • The update location specified in the registry of the target machines must be edited to point to the shared location path instead of the default Internet path.
  • Target users must have administrator privileges on their machines.

In addition, administrators must create the following files:

  1. Zero, one, or more than one Localized Windows Component Update file for each Locale that you want to support for a particular Windows Component Update. You do not need this file if you do not have a Localized Windows Component Update; if you do need this file, it is stored within the locale directory of that particular Windows Component Update.
  2. Zero, one, or more than one Main Windows Component Update file for each Windows Component Update you want to make available. You do not need this file if you do not have a Windows Component Update; if you do need this file, then you have to create this file and saved it within the Windows Component Update directory.
  3. Zero, one, or more than one Main Service Pack Update file for each Update you want to make available. You do not need this file if you do not have a Service Pack Update; if you do, this file has to be edited by you and saved within the appropriate Service Pack directory.
  4. One Visual Studio Update Session file. The server path points to this file and this file points to the other Service Pack Update files and/or Main Windows Component Update files. This file is mandatory and has to be created by you.

See section 8 for the Key value pairs and see section 9 for the Locale codes.

To create a Localized Windows Component Update file
  1. Copy the Service Release files to a shared location, such as a network share or Intranet site.
  2. Make sure that each locale version of Windows Component Update is located in its respective directory.

    For example, if you downloaded Windows Component Update in both English and Japanese, you would have two different locale versions of Windows Component Update. These locale versions must remain in separate directories and these directory names must use the locale code, in this case 1033 for English and 1041 for Japanese.

  3. Update each .vsupd file to point to the correct path and locale for .crc and .cab files.

    For example:

    crcpath=http://myupdate/vs7su1/1033/vs7sr.crc
    cabpath=http://myupdate/vs7su1/1033/vs7sr.cab
To create the Main Windows Component Update file
  1. Create a text file with the extension .vsupd and place the file within the Windows Component Update directory.
  2. Enter the following information in the file:

    [VSUPDATE]
    type=3
    identifier=<path to Windows Component Update directory>
    name=uniquename
    locale=1

    For example, if you have Windows Component Update 1 files in the directory vs7su1 with the English locale files in a subdirectory called 1033, you would include the following information in the .vsupd file:

    [VSUPDATE]
    type=3
    identifier=vs7su1
    name=Windows Component Update 1
    locale=1

    For more information on content in .vsupd files, see section 8.
To create the Main Service Pack Update file
  1. Browse to the directory containing the Service Pack files and open the file with the .vsupd file extension.
  2. Edit the following entries:

    Under [DOWNLOAD]

    cabpath=<path to Service Pack cab files>

    Note: If [Dependency_#] does not exist, skip the next section.

    Under [Dependency_#]:

    a. If TYPE=1 AND Locale=1 exist then:

    mapath=<path to the Main Windows Component Update file>

    b. If the TYPE=1 AND Locale=1 do not exist then:

    pdipath=<path to the Windows Component .pdi file>
    datpath=<path to the Windows Component .dat file>

    c. If TYPE=2 exists then:

    pdipath=<path to the Service pack .pdi file>
    datpath=<path to the Service pack. dat file>

    For example, if you have a shared location on the Intranet at http://myupdate and you decide to copy the Service Release that includes Service Pack 2 (which has a dependency vs7su1) to a directory called vs7sp_2 on that shared location, the Main Service Pack Update file would be as follows:

    [VSUPDATE]
    type=2
    identifier=vs7sp2
    name=Service Pack 2
    size=2072
    date=2001/03/22/6
    version=0000:11:1234
    key=CurrentSPLevel
    level=2

    [LAUNCH]
    type=3
    executable=STSP2.MSP
    arguments=REINSTALL=ALL REINSTALLMODE=omus REBOOT=ReallySuppress
    installtype=
    installstate=
    installlevel=
    installpackage=

    [DOWNLOAD]
    crcpath=
    cabpath=http://myupdate/vs7sp2/
    cabname=vs7sp2
    cabsize=500
    numcabs=5

    [DEPENDENCY_1]
    type=1
    identifier=vs7su1
    name=Test Windows Component Update 1
    locale=1
    mappath=http://myupdate/vs7su1/vs7su1.vsupd
To create the Visual Studio session file
  1. Create a text file with the extension .vsupd (e.g. vsupdate.vsupd) and save it in the shared location at the same level as the directories that have the Service Packs and Windows Component Updates.

    For example, http://myupdate/

  2. Add the necessary key values to the file. See section 8 for a list.

    In the following example, you have three Service Packs and two Windows Component Updates that you want to make available to your target users. A dependency exists between Service Pack 2 and Windows Component Update 1 and similarly a dependency exists between Service Pack 3 and Windows Component Update 2. To map these dependencies, the .vsud file would need to contain the following:

    [VSUPDATE]
    type=1
    vs7sp=3
    vs7su=2

    [VS7SP_1]
    mappath=http://myupdate/vs7sp1/vs7sp1.vsupd

    [VS7SP_2]
    mappath=http://myupdate/vs7sp2/vs7sp2.vsupd

    [VS7SP_3]
    mappath=http://myupdate/vs7sp3/vs7sp3.vsupd

    [VS7SU_1]
    mappath=http://myupdate/vs7su1/vs7su1.vsupd

    [VS7SU_2]
    mappath=http://myupdate/vs7su2/vs7su2.vsupd
To set the server path and disable Start Page notification on target machines
  1. Edit the registry on the client machines so that the server path points to the shared location on the network where the Visual Studio Update Session file is located.

    For example:

    Server Path=http://myupdate/vsupdate.vsupd< /P>

  2. To disable the Start Page notification on target machine, edit the registry on the client machine and change HKEY_CURRENT_USER\Software\Microsoft\VisualStudio\7.0\LastKnownUpdate to equal 9999.

  • Optional sections
